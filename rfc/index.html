Request for Comment (RFC)

RFC Process Flow

Product Manager/Owner —> Create RFC document —> Add Summary, Motivation, Guide-level explanation, Key personnel Principal Engineer —> Add Reference-level explanation —> Notify Architects if required —> Add Dependancies

RFC Template

The RFC should be clear on the desired outcomes, and concise in its length. A diagram saves a lot of time for the author and reader.

The minimum fields to required for RFC template

  • Summary: A paragraph on the explanation of the feature
  • Motivation: The reason why we’re doing this. The people who will benefit
  • Guide-level explanation: Explain the feature largely in terms of examples. How does the user flow. What does good look like.
  • Detailed Design: The technical proportion of the RFC. The proposal of implementation. Corner cases dissected by example
  • Architecture: If architectural changes required, add ADF link
  • Dependancies: Teams who need to be aware of the chance both upstream of downstream
  • Key personnel: A list of the key contacts, at minimum one member from Product and one from Engineering
  • Unanswered questions

RFC Comments

Comments around the RFC should be kept in limited where possible to two locations

Confluence: long-lived conversation Slack: casual conversation

Slack conversations should go into the channel <Product>-RFC. As casual conversations start moving into requirements, the discussion should move into Confluence
